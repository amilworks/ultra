# Napkin

- 2026-04-13: The Pro Mode `OpenAILike` seam was flexible enough for Bedrock-published gateways once we exposed `default_headers` and `default_query` in runtime config. API Gateway style publishes typically need `PRO_MODE_API_KEY_HEADER=X-API-Key` with a blank prefix rather than OpenAI bearer auth.
- 2026-04-13: Pro Mode had no dedicated `direct_response` execution branch in runtime, so conceptual Pro turns could be triaged correctly and still fall back into the normal tool-capable stack. The fix was to add an explicit fast-dialogue branch that uses the dedicated Pro Mode transport/model and to soften the tool-routing heuristic for conceptual discussions of methods like segmentation.
- 2026-04-13: Postgres locked up on production because its data directory lived on the barrel NFS mount (`/mnt/barrel-data/ultra/platform/postgres`). When the NFS server stalled, postgres backends entered uninterruptible `D` state and the app node started timing out on DB connections. Keep Postgres and Keycloak on local Docker volumes; use the barrel mount only for BisQue file/storage paths.
- 2026-04-13: The platform node should support direct SSH from the operator workstation. Prefer the direct platform-node path instead of assuming a jump-host flow.
- 2026-04-13: BisQue OIDC web login could succeed with a `tg_user` present but no persisted `BQUser`, leaving the admin Users tab empty and owner-scoped BisQue behavior inconsistent. The fix is to self-heal missing `BQUser` records during OIDC login and during bootstrap reconciliation for the seeded `admin` user, and to stop implying that local BisQue user creation is the right workflow when auth mode is OIDC-only.
- 2026-04-13: In the split-node production topology, BisQue admin AJAX uses `/admin/*`, so that prefix must be included in both the public app-node BisQue matcher and the internal platform-node proxy matcher. Fixing only one hop still leaves the admin UI broken.
- 2026-04-13: The running `bisque-server` container is importing Python and static admin assets from `/usr/lib/bisque/lib/python3.10/site-packages/bq/...`, not the checked-out `/source/...` tree. Hotpatching `/source` alone does not change live behavior; either rebuild cleanly or patch the installed paths and restart `bisque-server`.
- 2026-04-14: When running `kcadm.sh` through `ssh ... docker exec ... sh -lc`, escape container env vars like `\\$KEYCLOAK_ADMIN` so the host shell does not eat them before they reach the container. An empty expansion looks like a Keycloak credential failure even when the container env is correct.
- 2026-04-14: The local Pro Mode UI can show a plausible direct-response draft even when the Bedrock-published Opus transport actually failed. For this path, trust `metadata.pro_mode.runtime_status` and the published conversation/message ids, not just the visible answer text or the configured model label.
- 2026-04-14: The UCSB Bedrock publish is healthy on `/api`, but the host root still returns `403`, so probes must hit `...execute-api.../api/health` and `.../api/conversation`. A tiny typo in the local ignored `.env` API key was enough to make the whole local Opus path look broken even though the gateway itself was fine.
- 2026-04-15: The production app node can silently fall back to `gpt-oss-120b` even when all Opus env fields look populated if the published API key is off by just a couple characters. Verify the exact `PRO_MODE_API_KEY` contents, not just presence/length, and avoid `source /etc/ultra/ultra-backend.env` for ad hoc checks when values contain spaces like `BISQUE_AUTH_OIDC_SCOPE=openid profile email`.
- 2026-04-14: BisQue already supports zero-copy registration of existing files through `blob_service/paths/insert` and `import/insert_inplace`, but only for paths that match a configured store/mount. For NFS-backed data, mount the export on the BisQue server and configure it as a `file://` store; the normal `/import/transfer` upload flow still copies bytes.
- 2026-04-14: The Megaseg wrapper must launch the cyto-dl virtualenv by its venv path (`.../venv/bin/python`) without resolving the symlink to the base interpreter. Resolving the path strips the virtualenv site-packages and makes imports like `numpy` appear missing even though the venv itself is healthy.
- 2026-04-14: Anonymous `/v1/chat` follow-up requests only recover prior run state and `latest_result_refs` if the client preserves the anonymous session cookie. Reusing `conversation_id` alone is not enough for curl-based live tests; use a cookie jar or pass the artifact paths explicitly.

- 2026-04-14: Agno native Bedrock Claude requires real AWS IAM/session auth; UCSB published API key endpoint is a separate API Gateway wrapper and returned 403 on exact documented /health and /conversation probes. Prefer native aws_bedrock_claude transport for local Opus testing when AWS creds/profile are configured.
- 2026-04-15: On this machine, shell snippets that rely on `mapfile` are not portable enough even when the login shell says `bash`; use `rg -0 | xargs -0 ...` for repo-wide rewrite passes instead.
- 2026-04-15: Release scanners can false-positive on their own regex definitions. Exclude the scanner script itself or keep the patterns in a separate data file before trusting a “dirty” scan result.
- 2026-04-15: Frontend run-step UI should never render backend orchestration names directly. Keep user-facing progress copy in a shared display-layer map so labels, subtitles, and fallback thinking text stay polished and external-facing.
- 2026-04-15: In a split-node production setup, keep frontend and app backends on the public app node and keep BisQue, Keycloak, and Postgres on the platform node. Ordinary app rollouts should target the app node; platform work should target the platform node.
- 2026-04-15: `scripts/deploy_ultra_frontend.sh` must support the real web server on the host. If the app node serves traffic with `Caddy` instead of `nginx`, the deploy helper should validate `/etc/caddy/Caddyfile` and reload `caddy`.
- 2026-04-15: For production app rollouts, deploy the exact committed SHA snapshot, not the current dirty tree. Building from `git archive <sha>` avoids leaking unrelated local edits into the release while still letting the workspace stay dirty.
- 2026-04-15: Pro Mode strict tool routing needs model-specific segmentation keywords too. If `_is_segmentation_request` misses names like `MegaSeg` or `DynUNet`, the runtime silently drops out of strict segmentation workflows, and strict completion checks must require the full Megaseg-plus-quantification tool set rather than accepting a partial run.
- 2026-04-15: When restyling the prompt composer, remember that `PromptInput` always brings the `pk-prompt-input` base class; if the app-specific card class does not explicitly zero out its padding, the old shell spacing stacks underneath the new design and makes the composer feel subtly wrong.
- 2026-04-15: The chat shell needs a real viewport height constraint, not just `min-height`, or the whole document starts scrolling and `use-stick-to-bottom` can no longer manage the transcript. For per-chat restore, save scroll state on real user scroll and block stale writes during conversation swaps so the outgoing chat does not get overwritten by the incoming chat’s layout pass.
- 2026-04-15: For `use-stick-to-bottom` debugging, synthetic `scrollTop` mutations can give false confidence. Verify transcript restore with real wheel/touch-style scrolling in the browser, because the hook’s lock state is tied to user-driven scroll behavior.
- 2026-04-15: On iPhone Safari, do not design a web-app chat drawer around an edge-swipe-to-open gesture. The browser already owns that edge for back navigation, so the safer premium-mobile pattern is a wide left sheet opened by a clear tap target, closed by overlay tap or chat selection, with `16px` inputs to avoid focus zoom.
- 2026-04-15: A repeatable mobile smoke check is worth keeping even for a Vite-only frontend. Verifying `scrollWidth === innerWidth` plus drawer width, mobile font sizes, and drawer-close-on-selection at `320`, `390`, and `430` catches the most common phone regressions before they make it back into the UI.
- 2026-04-15: Mobile drawer rows cannot reuse desktop `justify-between` layouts when the right-side shortcut hint is hidden. In Safari this makes simple icon-label rows look broken and stretched. Group the icon + label into a left cluster and validate the final drawer in the iPhone simulator, not just Playwright screenshots.
- 2026-04-15: For split-node app deploys, `/mnt/barrel-data/ultra/releases` is a remote app-node concern, not a local mount on the workstation. Build the exact commit snapshot locally, then stage it onto the app node before running the release scripts there.
- 2026-04-15: When staging a production app release, verify the remote release tree actually contains `backend/scripts/deploy_ultra_*.sh` before invoking deploy. A partial or interrupted rsync can leave the directory structure in place while critical files are still missing.
- 2026-04-15: `PLATFORM_DATA_ROOT=/mnt/barrel-data/ultra/platform` is only correct if the platform node actually mounts the intended private NFS export on `/mnt/barrel-data`. A local directory with the right name silently puts BisQue uploads/generated data on local disk instead of shared storage. Verify with `mount` or `df`, not just the env file, and keep only BisQue file paths on shared storage while Postgres/Keycloak stay on local Docker volumes.
- 2026-04-15: Pro Mode image-tool routing cannot rely only on uploaded files or selection context. If a user pastes a real scientific image path into the prompt, the runtime must still treat that as a direct image target or it can demote a Megaseg request into open-ended reasoning and hallucinate downstream mask paths.
- 2026-04-15: Tool execution needs the same prompt-path awareness as routing. Strict fallback execution can recover a skipped Megaseg call only if `execute_tool_call` can infer `file_path`/`file_paths` from prompt-scoped local or `s3://...` image paths when the model leaves arguments blank.
- 2026-04-15: Public Allen Cell `s3://allencell/...` microscopy inputs work through BioIO and Megaseg with anonymous S3 fallback, but the reader stack may log credential errors before recovering with `anon=True`. Treat those warnings as recoverable unless the final load actually fails.
- 2026-04-15: Large public OME-Zarr inputs can validate and start Megaseg inference correctly on this workstation, but CPU throughput is much slower than smaller TIFF cases because the runner currently drops to the first timepoint and then processes a large 2D field. Keep a small TIFF smoke case around for fast end-to-end checks.
- 2026-04-15: Pro Mode research-program heuristics must not treat generic conceptual mentions of `image`, `images`, or `segmentation` as proof of a data-backed analysis task. Otherwise theory/methods prompts about microscopy silently drop from the Opus reasoning path into the `gpt-oss-120b` validated-tool path.
- 2026-04-15: The current UCSB `bedrock_published_api` Opus transport only exposes a coarse `enableReasoning` boolean. Agno’s native Claude transports already support thinking budgets plus `max_tokens`, `temperature`, `top_p`, and `top_k`; tune quality there instead of assuming OpenAI-compatible `reasoning_effort` will change Anthropic behavior through the published wrapper.
- 2026-04-15: Agno’s AWS Claude docs currently advertise Bedrock API-key auth, but the shipped `agno` AWS Claude wrapper still raises on `AWS_BEDROCK_API_KEY`. The underlying `anthropic.lib.bedrock.AnthropicBedrock` client does accept bearer-token auth via `api_key` / `AWS_BEARER_TOKEN_BEDROCK`, so if native Bedrock bearer auth matters, inject prebuilt Anthropic Bedrock clients or wait for Agno to remove the guard.
- 2026-04-15: The current production `PRO_MODE_API_KEY` on the app node is an API Gateway key for the published `execute-api ... /api` wrapper, not a native Bedrock bearer token. A direct `AnthropicBedrock(api_key=...)` probe failed with `403 Invalid API Key format`, so do not flip `PRO_MODE_TRANSPORT=aws_bedrock_claude` until real Bedrock bearer or IAM credentials exist on the app node.
- 2026-04-15: A clean release worktree is only safe if every runtime dependency is actually in the committed snapshot. I tried staging a minimal Pro Mode patch from a clean commit-only tree and one backend instance crash-looped on a missing `src.tools.extract_scientific_image_paths_from_text` import, which meant production behavior had drifted beyond that commit. After any “exact SHA” staging, check both app ports (`127.0.0.1:8001` and `:8002`) and the public health endpoint before assuming the rollout is healthy.
- 2026-04-16: For public-release audits, the useful split is: current operator docs and changed files must be scrubbed of private hostnames, usernames, and production URLs, while the older BisQue tree still contains many UCSB-branded public or historical references that are noisy but not necessarily secret. Keep the normal release scanner focused on true operator leakage, and track the legacy branding cleanup separately.
- 2026-04-16: Megaseg runtime resolution must read from `get_settings()` as well as raw process env. Otherwise `.env`/settings-backed values like `MEGASEG_CHECKPOINT_PATH` or `MEGASEG_PYTHON` can look configured in the app but still be invisible to the live tool path, especially under local dev helpers that do not source `.env`.
- 2026-04-16: The Megaseg runner already supports single-channel `ZYX` microscopy volumes if it receives `structure_channel=<index_base>` and `nucleus_channel=None`, but the top-level tool defaults (4/6) still cause immediate channel-range failures on uploaded single-channel TIFFs unless the tool auto-normalizes those arguments before launch.
- 2026-04-16: Keep `src.science.__init__` lazy. Eager package-level imports pulled backend-only settings dependencies like `pydantic_settings` into the standalone Megaseg GPU service and broke service startup even though the Megaseg core itself was lightweight.
- 2026-04-16: FastAPI multipart job intake needs `python-multipart` installed explicitly. JSON-only service tests will not catch this; at least one regression test must exercise a real multipart upload path.
- 2026-04-16: Remote Megaseg service support for local `.ome.zarr` uploads needs directory-store transport, not plain file upload. Packaging the directory as `.tar.gz` on the client and safely unpacking it server-side preserves the BioIO-compatible path shape.
- 2026-04-16: The GPU-node OME-Zarr reader stack broke with `numcodecs 0.16.5` because `zarr 2.18.3` still imports `numcodecs.blosc.cbuffer_sizes`. Pin `numcodecs==0.13.1` (with `numpy==1.26.4`) in the Megaseg service env to match the working local stack.
- 2026-04-16: Repo diligence found a real product-security smell: the frontend stores the orchestrator API key in `localStorage` and also appends it as `api_key` on URL-based resource/auth endpoints. For enterprise-readiness reviews, treat that pattern as a blocking auth hardening item, not a cosmetic cleanup.
- 2026-04-16: Megaseg chat cards cannot rely on raw progress-event artifact paths for preview URLs. The tool emits absolute output paths like `.../megaseg_overlay_mip.png`, but the browser download route only serves registered run artifacts such as `tool_outputs/<hash>__...`. Prefer hydrated artifact records (or fetch `/v1/artifacts/{run_id}` first) before building preview links.
- 2026-04-16: If a tool card depends on post-hydration metadata, include that metadata in the `useMemo` dependency list. I initially hydrated Megaseg summary JSON into `message.responseMetadata`, but the card stayed stale because `buildToolResultCards(...)` only depended on `progressEvents` and `runArtifacts`.
- 2026-04-16: A generalized research-digest UI is only as good as the backend response contract feeding it. If normal Agno/Pro Mode answers do not emit a contract, the frontend can render but cannot reliably prioritize findings, measurements, or next steps.
- 2026-04-16: For digest lead text, prefer explicit takeaway markers like `Bottom line:`, `Primary suspect:`, or `Most likely failure mode:` before falling back to “first paragraph” heuristics. Otherwise markdown-heavy scientific answers collapse into long, unreadable digest summaries.
- 2026-04-16: When hardening auth in `src/api/main.py`, do not capture `ALLOW_QUERY_API_KEY_COMPAT` or `ORCHESTRATOR_API_KEY` into create-app locals if tests or staged deploys may patch settings. Read them at request time so compat behavior stays observable and rollout flags do what operators expect.
- 2026-04-16: Frontend smoke automation should not assume `127.0.0.1:8000` or `:5173` are free. Use isolated high ports in the wrapper and pass them through the Vite proxy target plus smoke URL env vars, or the check will flake as soon as a local dev stack is already running.
- 2026-04-16: `src.tools` needs the local `bqapi` path bootstrap before first-party imports that transitively rely on BisQue modules, but doing that as a raw function call before imports breaks Ruff’s import-order contract. The clean fix is a dedicated bootstrap module imported as part of the first-party import group, not a top-of-file side-effect statement.
- 2026-04-16: Tool schemas now have a cleaner ownership boundary in `src.tooling.domains`. High-level orchestration code like `src.llm` should import tool schema constants from that package and keep `src.tools` focused on runtime execution helpers, rather than relying on the old god-file as a re-export layer.
- 2026-04-16: Frontier-style scientific presentation works better when evidence UI is appendix-like rather than headline-like. Keep structured evidence blocks for quantitatively grounded tool turns, place them after the prose, and avoid glossy summary cards or generic confidence badges on simple answers.
- 2026-04-16: Shiki’s fine-grained `shiki/langs/*` and `shiki/themes/*` loader paths work well for bundle control in this frontend, but the current TS bundler config does not pick up their declarations automatically. Keep the local `frontend/src/shiki-loader-modules.d.ts` shim or the build will fail even though the runtime imports are valid.
- 2026-04-16: Broad Rollup chunk rules can quietly undo a lazy-loading refactor. I initially matched every `/shiki/` module into one manual chunk, which collapsed the fine-grained language/theme split back into a giant 1.2 MB highlight runtime. For this repo, targeted vendor chunking is safer than catch-all rules for libraries that already code-split internally.
- 2026-04-16: The current honest frontend budget shape is: main shell, shared UI vendor, markdown runtime, three/viewer runtime, upload viewer slice, and a lazy Shiki bootstrap chunk. The React compiler and hooks plugin still surface many warnings in `App.tsx` and the HDF5/viewer stack, so treat this as a ratcheted floor, not the end-state.
- 2026-04-16: Local workflow simulation can be skewed by the ignored repo `.env`. In this checkout, `LOG_FORMAT=json` made backend tests fail at import time even though the GitHub workflow would run clean. For accurate local CI-equivalent runs, either neutralize those overrides explicitly or keep local-only env conventions compatible with import-time code paths.
- 2026-04-16: `deploy/env/platform.env.example` needs to be complete enough for `docker compose --env-file ... config` to pass. Missing required placeholders like `BISQUE_AUTH_COOKIE_SECRET` silently undermine the “production docs are self-consistent” story even when the application code is fine.
