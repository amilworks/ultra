# Napkin

- 2026-04-13: The Pro Mode `OpenAILike` seam was flexible enough for Bedrock-published gateways once we exposed `default_headers` and `default_query` in runtime config. API Gateway style publishes typically need `PRO_MODE_API_KEY_HEADER=X-API-Key` with a blank prefix rather than OpenAI bearer auth.
- 2026-04-13: Pro Mode had no dedicated `direct_response` execution branch in runtime, so conceptual Pro turns could be triaged correctly and still fall back into the normal tool-capable stack. The fix was to add an explicit fast-dialogue branch that uses the dedicated Pro Mode transport/model and to soften the tool-routing heuristic for conceptual discussions of methods like segmentation.
- 2026-04-13: Postgres locked up on production because its data directory lived on the barrel NFS mount (`/mnt/barrel-data/ultra/platform/postgres`). When the NFS server stalled, postgres backends entered uninterruptible `D` state and the app node started timing out on DB connections. Keep Postgres and Keycloak on local Docker volumes; use the barrel mount only for BisQue file/storage paths.
- 2026-04-13: `nail04` now supports direct SSH as `amil@nail04.ece.ucsb.edu`, and it is the right path for platform deploy/debug work instead of assuming the older jump-host flow.
- 2026-04-13: BisQue OIDC web login could succeed with a `tg_user` present but no persisted `BQUser`, leaving the admin Users tab empty and owner-scoped BisQue behavior inconsistent. The fix is to self-heal missing `BQUser` records during OIDC login and during bootstrap reconciliation for the seeded `admin` user, and to stop implying that local BisQue user creation is the right workflow when auth mode is OIDC-only.
- 2026-04-13: In the split-node production topology, BisQue admin AJAX uses `/admin/*`, so that prefix must be included in both the public `nail01` Caddy BisQue matcher and the internal `nail04` platform Caddy matcher. Fixing only one hop still leaves the admin UI broken.
- 2026-04-13: The running `bisque-server` container is importing Python and static admin assets from `/usr/lib/bisque/lib/python3.10/site-packages/bq/...`, not the checked-out `/source/...` tree. Hotpatching `/source` alone does not change live behavior; either rebuild cleanly or patch the installed paths and restart `bisque-server`.
- 2026-04-14: When running `kcadm.sh` through `ssh ... docker exec ... sh -lc`, escape container env vars like `\\$KEYCLOAK_ADMIN` so the host shell does not eat them before they reach the container. An empty expansion looks like a Keycloak credential failure even when the container env is correct.
- 2026-04-14: The local Pro Mode UI can show a plausible direct-response draft even when the Bedrock-published Opus transport actually failed. For this path, trust `metadata.pro_mode.runtime_status` and the published conversation/message ids, not just the visible answer text or the configured model label.
- 2026-04-14: The UCSB Bedrock publish is healthy on `/api`, but the host root still returns `403`, so probes must hit `...execute-api.../api/health` and `.../api/conversation`. A tiny typo in the local ignored `.env` API key was enough to make the whole local Opus path look broken even though the gateway itself was fine.
- 2026-04-15: The production app node can silently fall back to `gpt-oss-120b` even when all Opus env fields look populated if the published API key is off by just a couple characters. Verify the exact `PRO_MODE_API_KEY` contents, not just presence/length, and avoid `source /etc/ultra/ultra-backend.env` for ad hoc checks when values contain spaces like `BISQUE_AUTH_OIDC_SCOPE=openid profile email`.
- 2026-04-14: BisQue already supports zero-copy registration of existing files through `blob_service/paths/insert` and `import/insert_inplace`, but only for paths that match a configured store/mount. For NFS-backed data, mount the export on the BisQue server and configure it as a `file://` store; the normal `/import/transfer` upload flow still copies bytes.
- 2026-04-14: The Megaseg wrapper must launch the cyto-dl virtualenv by its venv path (`.../venv/bin/python`) without resolving the symlink to the base interpreter. Resolving the path strips the virtualenv site-packages and makes imports like `numpy` appear missing even though the venv itself is healthy.
- 2026-04-14: Anonymous `/v1/chat` follow-up requests only recover prior run state and `latest_result_refs` if the client preserves the anonymous session cookie. Reusing `conversation_id` alone is not enough for curl-based live tests; use a cookie jar or pass the artifact paths explicitly.

- 2026-04-14: Agno native Bedrock Claude requires real AWS IAM/session auth; UCSB published API key endpoint is a separate API Gateway wrapper and returned 403 on exact documented /health and /conversation probes. Prefer native aws_bedrock_claude transport for local Opus testing when AWS creds/profile are configured.
- 2026-04-15: On this machine, shell snippets that rely on `mapfile` are not portable enough even when the login shell says `bash`; use `rg -0 | xargs -0 ...` for repo-wide rewrite passes instead.
- 2026-04-15: Release scanners can false-positive on their own regex definitions. Exclude the scanner script itself or keep the patterns in a separate data file before trusting a “dirty” scan result.
- 2026-04-15: Frontend run-step UI should never render backend orchestration names directly. Keep user-facing progress copy in a shared display-layer map so labels, subtitles, and fallback thinking text stay polished and external-facing.
- 2026-04-15: The current UCSB production split is `nail01` for the public edge/frontend/app backends and `nail04` for the BisQue/Keycloak/Postgres platform stack. Ordinary app rollouts should target `ssh bisque_amil@nail01.ece.ucsb.edu`; ordinary platform rollouts should target `ssh amil@nail04.ece.ucsb.edu`.
- 2026-04-15: `scripts/deploy_ultra_frontend.sh` must support the real web server on the host. `nail01` serves the app with `Caddy`, not `nginx`, so the deploy helper should validate `/etc/caddy/Caddyfile` and reload `caddy`.
- 2026-04-15: For production app rollouts, deploy the exact committed SHA snapshot, not the current dirty tree. Building from `git archive <sha>` avoids leaking unrelated local edits into the release while still letting the workspace stay dirty.
- 2026-04-15: Pro Mode strict tool routing needs model-specific segmentation keywords too. If `_is_segmentation_request` misses names like `MegaSeg` or `DynUNet`, the runtime silently drops out of strict segmentation workflows, and strict completion checks must require the full Megaseg-plus-quantification tool set rather than accepting a partial run.
